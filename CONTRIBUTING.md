# Contributing to eslint-plugin-todo-tickets

Thank you for considering contributing! Please follow these steps to help us keep the project healthy and maintainable.

## 1. Check existing issues first

Before opening a new issue:

- Search the [issues list](https://github.com/Fcmam5/eslint-plugin-todo-tickets/issues) to see if someone has already reported the same problem or proposed the same feature.
- If you find an existing issue, add a reaction or comment to show your interest.

## 2. Open an issue before opening a PR

We prefer discussing changes in an issue first. This helps:

- Avoid duplicate work.
- Confirm the approach is aligned with the project's goals.
- Keep PRs focused and easier to review.

If you plan to contribute code, please open an issue describing:

- The problem you want to solve.
- The proposed solution.
- Any edge cases you considered.

## 3. AI/LLM-generated contributions are not welcome

Automated contributions without human review are not accepted. If you are using an LLM agent:

- Do not open PRs directly.
- Instead, write a detailed issue with:
  - Clear problem description.
  - Expected behavior.
  - Test cases (preferably).
  - Steps to reproduce.

Let a human pick up the task and implement it thoughtfully.

## 4. No "AI slop"

We expect contributors to:

- Review and understand the code they are changing.
- Test their changes manually and with the existing test suite.
- Write clear, maintainable code.
- Add or update tests where appropriate.

If you use AI tools to assist, treat the output as a draft. You are responsible for verifying correctness, style, and suitability before submitting.

## 5. Development workflow

1. Fork the repository.
2. Create a branch for your work.
3. Install dependencies: `npm install`.
4. Run tests: `npm test`.
5. Run linting: `npm run lint`.
6. Make your changes.
7. Ensure tests pass and linting succeeds.
8. Open a PR with a clear description and link to the related issue.
